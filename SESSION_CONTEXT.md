# Контекст сессии — для продолжения работы

Этот файл — саммари того, что произошло в сессии разработки, чтобы новая
сессия (или другой человек) могла продолжить без повторного объяснения
контекста. Продуктовые решения — в [PROJECT_SPEC.md](PROJECT_SPEC.md),
инструкция по использованию — в [README.md](README.md), деплой — в
[DEPLOYMENT.md](DEPLOYMENT.md). Здесь — то, что не попало ни туда, ни туда:
как это стало кодом и какие решения были приняты уже в процессе реализации.

## Как появился проект

Идея прошла через интервью-«грилинг» (см. PROJECT_SPEC.md — там пошагово
зафиксированы все продуктовые решения: 50+ собственных каналов, цель —
оживление обсуждений, мульти-аккаунтность с прокси, персоны/тон, подписи с
точечным и массовым редактированием, диапазоны количества комментаторов и
задержки, суточные лимиты, автостатусы аккаунтов, контент-фильтр,
веб-панель, платный ИИ-API). После согласования спецификации пользователь
попросил сразу приступить к реализации.

## Состояние на конец сессии

**Полностью реализовано и локально smoke-тестировано** (запускался uvicorn
на SQLite, руками проверены все страницы панели и основные POST-сценарии:
добавление аккаунта, массовая/точечная подпись, изменение настроек, toggle
статуса). **Не тестировалось** — реальное подключение к Telegram (нет живых
API ID/Hash/session-строк в этой сессии), реальные вызовы ИИ-провайдера,
полный цикл «новый пост → комментарий» с настоящими аккаунтами. Это первое,
что стоит проверить на VPS после деплоя.

## Стек и почему так

- **Python 3.12 + FastAPI + Jinja2** для панели — без отдельного JS-фреймворка
  и без Node.js вообще (на машине разработки его не было, да и для
  табличного CRUD-инструмента для одного пользователя SPA — избыточность).
  Формы шлют обычные POST + redirect, без htmx/AJAX.
- **Telethon** — единственная зрелая Python-библиотека для MTProto
  (пользовательские аккаунты, не Bot API — это принципиально, т.к. боты не
  могут комментировать в чужом/своём канале как обычные подписчики).
- **PostgreSQL + SQLAlchemy 2.0 async + Alembic** — стандартный выбор,
  осмысленный при десятках аккаунтов и постоянной записи в журнал.
- **APScheduler (in-memory, без persistent jobstore)** — для отложенной
  публикации комментариев (диапазон задержки). Осознанное упрощение: при
  рестарте worker уже запланированные (но ещё не опубликованные) задания
  теряются. Задокументировано как известное ограничение в README. Если это
  станет проблемой — решение: `SQLAlchemyJobStore` из apscheduler.
- **Anthropic/OpenAI как провайдеры** — абстракция в
  `app/services/ai_generator.py`, переключается переменной `AI_PROVIDER`.
  Конкретная модель не была принципиальна пользователю, оставлена на моё
  усмотрение (Claude Haiku по умолчанию — короткие комментарии не требуют
  топовой модели).
- **Разделение web/worker на два процесса** (два контейнера в
  docker-compose) — веб-панель не держит постоянных Telegram-соединений,
  этим занимается только worker. Синхронизация между ними — **не через
  прямой вызов** (это разные процессы), а через **периодический опрос БД**
  worker'ом каждые 60 секунд (`app/services/sync.py`). Изменения в панели
  (новый аккаунт, новое закрепление за каналом) подхватываются worker'ом в
  течение минуты без рестарта. Это сознательный выбор в пользу простоты
  вместо шины сообщений (Redis pub/sub и т.п.) — оправдан тем, что для
  личного инструмента задержка в 60 секунд не критична.

## Нетривиальные решения, принятые по ходу кода

- **Watcher-паттерн для детекта новых постов**: вместо того чтобы каждый
  закреплённый за каналом аккаунт слушал этот канал (дублирующиеся события),
  на каждый канал назначается **один** «слушающий» аккаунт
  (`TelegramManager.set_watcher`, логика выбора — в `sync.py`: предпочтение
  отдаётся ACTIVE-аккаунтам, а не LIMITED). Публикует комментарий при этом
  тот аккаунт, который был выбран оркестратором — независимо от того, кто
  «услышал» пост.
- **Хранение id канала**: используется `get_peer_id()` из Telethon (marked
  id, вида `-100xxxxxxxxxx`) — это то, что Telethon сам рекомендует хранить
  для стабильного кросс-сессионного обращения к сущности, а не «сырой» id.
- **Резолвинг/джойн канала из веб-процесса**: у web нет постоянных
  подключений (они только у worker), поэтому при добавлении канала или
  закреплении аккаунта веб-процесс поднимает **одноразовое** Telethon-
  подключение через уже сохранённый аккаунт (`resolve_channel_standalone`,
  `join_channel_standalone` в `telegram_manager.py`) и сразу его закрывает.
- **Авто-джойн при закреплении**: когда аккаунт закрепляют за каналом через
  панель, сервис сам пытается вступить в канал этим аккаунтом (см.
  `POST /accounts/{id}/assignments` в `app/web/routes/accounts.py`). Если
  вступление не удалось — закрепление всё равно сохраняется, ошибка
  показывается во flash-сообщении (не блокирует сохранение).
- **`ondelete="SET NULL"` на `accounts.persona_id`** — чтобы удаление
  кастомной персоны не падало с ошибкой внешнего ключа, если она уже
  назначена на какой-то аккаунт (просто обнуляется).
- **Starlette API gotcha**: в установленной версии (`starlette==1.3.1`)
  сигнатура `TemplateResponse` — `(request, name, context)`, а не старый
  `(name, {"request": request, ...})`. Это не задокументированное явно
  расхождение с более старыми туториалами — если при апгрейде зависимостей
  снова вылезет `TypeError: unhashable type: 'dict'` в шаблонах, дело в этом.
- **`DetachedInstanceError` в шаблонах**: любой relationship, к которому
  обращается Jinja-шаблон (например `account.persona.name`), должен быть
  eager-loaded (`joinedload`/`selectinload`) в запросе — сессия SQLAlchemy
  закрывается до рендеринга шаблона. Уже поймано и исправлено для
  `Account.persona` в списке аккаунтов; если будете добавлять новые поля в
  шаблоны — держите это в уме.
- **Встроенные персоны** (`app/services/seed.py`) — 5 пресетов тона
  (нейтральный, саркастичный, экспертный, эмоциональный, лаконичный),
  идемпотентно досеваются при старте и web, и worker процессов.

## Известный пробел, требующий решения

**У веб-панели нет авторизации.** Это не было явно проговорено на этапе
интервью (стоило спросить отдельно). Сейчас риск закрыт эксплуатационно —
в DEPLOYMENT.md порт 8000 не открывается наружу, доступ только через
SSH-туннель. Если понадобится публичный доступ без туннеля — нужно либо
добавить basic auth/логин в само приложение, либо nginx с Basic Auth и
HTTPS перед ним. Ни то, ни другое пока не реализовано.

## Структура кода (куда что класть дальше)

```
app/
  config.py, db.py, crypto.py       — настройки, БД, шифрование секретов
  models/                            — SQLAlchemy-модели (6 таблиц)
  services/
    telegram_manager.py              — вся работа с Telethon (плюс standalone-хелперы для веб-процесса)
    orchestrator.py                  — бизнес-логика: кто комментирует, когда, как
    sync.py                          — сверка воркера с БД (коннекты, watcher'ы)
    ai_generator.py                  — Claude/OpenAI абстракция
    content_filter.py                — стоп-слова
    notifier.py                      — уведомления владельцу
    seed.py                          — встроенные персоны
  web/routes/                        — по одному файлу на раздел панели
  web/templates/                     — Jinja2, наследуют base.html
  worker.py                          — точка входа воркера
  main.py                            — точка входа веб-панели (FastAPI)
scripts/generate_session.py          — CLI для получения session-строки аккаунта
migrations/                          — Alembic, одна начальная миграция
```
